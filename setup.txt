1. How This Document Fits
The Build Specification v2.0 defines what to build -- features, data model, API surface, phase boundaries. This document answers the practical question that one assumes but doesn't spell out: what to install, which accounts to open, and how to lay the repo out before any of that gets written.
SAME GROUND RULES APPLY Phase 1 scope only. One deployable service, one repo. Where the Build Specification is silent and a choice had to be made below, it's marked as a recommendation, not a requirement -- swap it for whatever the build agent or developer already knows well.

5. Environment Variables
Secrets stay out of the repo, per Section 5 of the Build Specification -- environment variables only. Backend example below; fill in real values per environment and never commit this file with real secrets in it.
# backend/.env.example NODE_ENV=development PORT=3000 DATABASE_URL=postgres://user:password@host:5432/adrash
JWT_ACCESS_SECRET=replace-me JWT_REFRESH_SECRET=replace-me JWT_ACCESS_EXPIRES_IN=15m JWT_REFRESH_EXPIRES_IN=30d
OTP_CODE_TTL_SECONDS=300 OTP_MAX_REQUESTS_PER_HOUR=5
FCM_PROJECT_ID=replace-me FCM_SERVICE_ACCOUNT_JSON=./secrets/fcm-service-account.json
SMS_PROVIDER_API_KEY=replace-me SMS_PROVIDER_SENDER_ID=replace-me
Mobile side: Expo reads configuration through app.config.js and EAS secrets rather than a .env file. At minimum it needs the API base URL and the EAS project ID -- both fine as plain config, neither is a secret.

7. Project and Repo Structure
The spec calls for one deployable service and one repo, not multiple repos and not a service mesh. That gives a single repo holding the backend service and the mobile client as separate top-level folders. "One deployable service" refers to the backend; the mobile app is inherently a separate build artifact (an app binary), but it isn't a second service and it doesn't need its own repo.

8. What This Document Does Not Cover
Same phase boundary as the Build Specification. None of the following gets installed, provisioned, or paid for until its phase is reached:
- Redis, a message queue, or WebSockets -- Phase 2. No account, package, or docker-compose entry for these yet.
- Chapa, telebirr, CBE Birr, or any payments SDK -- Phase 2. - An automated ID/KYC verification service -- Phase 2; Phase 1 is a manual review queue, and that needs
no new account. - PostGIS or any geospatial extension -- Phase 3, and only past the trigger in the spec. - Any monitoring or alerting service, container registry, or orchestration platform -- Phase 3, gated behind the
same triggers as the Build Specification.
